📞 System Calls 💡 Catatan
System Call itu seperti "menu pesanan" ke OS. Program bilang: "Hei OS, tolong bukakan file ini buat saya!" dan OS yang bertugas menjalankannya dengan aman. Programmer jarang pakai system call langsung — biasanya lewat API seperti POSIX (Linux/Mac) atau Win32 API (Windows).
System Call = antarmuka pemrograman untuk layanan OS.

Cara kirim parameter:
📦 Register
Parameter dimasukkan ke register CPU. Paling cepat, tapi terbatas jumlahnya.
🗃️ Blok Memori
Alamat blok memori dimasukkan ke register. Cocok untuk parameter banyak.
📚 Stack
Parameter di-push ke stack, OS yang pop. Fleksibel untuk berbagai jumlah parameter.

Thread & Concurrency
Threads, Multicore, Implicit Threading

Thread itu seperti "pekerja dalam satu toko." Satu proses bisa punya banyak thread yang bekerja bersamaan, berbagi toko (memori) tapi punya tugas masing-masing. 🧵

🧵 Konsep Thread
Thread = unit terkecil utilisasi CPU. Dalam satu proses, thread berbagi:
Code Data Files — tetapi memiliki sendiri: Stack Register Program Counter
≋ ≋ ≋
Paralelisme = menjalankan >1 tugas betul-betul bersamaan (butuh multi-core).
Konkurensi = mendukung >1 tugas "membuat progres" — bisa dengan 1 core lewat penjadwalan.

🗂️ Model Multithreading
Many-to-One
Banyak user thread → 1 kernel thread. Jika 1 thread blocking, semua block.
One-to-One ⭐
1 user thread → 1 kernel thread. Paling umum! (Linux, Windows). Lebih paralel.
Many-to-Many
Banyak user thread → banyak (≤) kernel thread. Fleksibel, rumit diimplementasi.
👆 Klik model untuk penjelasan visual!
🤖 Implicit Threading
Thread dibuat & dikelola oleh compiler atau runtime library, bukan programmer secara manual.
Thread Pools▶
Sekumpulan thread siap pakai. Tugas baru langsung diambil thread idle. Hemat overhead pembuatan thread.
Fork-Join▶
Tugas besar di-fork jadi sub-tugas paralel, lalu di-join untuk hasil akhir. Seperti divide-and-conquer.
OpenMP▶
Direktif compiler untuk C/C++/Fortran. Cukup tambah #pragma omp parallel dan OpenMP urus threading-nya.
Grand Central Dispatch▶
Apple's framework untuk macOS/iOS. Tugas dimasukkan ke dispatch queue, sistem atur thread-nya otomatis.

Sinkronisasi Proses
Race Condition, Mutex, Semaphore, Masalah Klasik

Bayangkan 2 orang menulis di papan tulis yang sama secara bersamaan — hasilnya kacau! Sinkronisasi mencegah "tabrakan" ini pada proses yang berbagi data. 🚦

🏁 Race Condition & Critical Section
Race Condition = beberapa proses akses & manipulasi data bersama secara bersamaan → hasil tergantung urutan eksekusi.

Critical Section = bagian kode yang mengakses data bersama. Hanya boleh 1 proses pada satu waktu!
~ ~ ~
Syarat Solusi Critical Section:
  • Mutual Exclusion — hanya 1 proses dalam CS sekaligus
  • Progress — kalau CS kosong, keputusan siapa masuk tidak boleh ditunda selamanya
  • Bounded Waiting — ada batas berapa kali proses lain boleh masuk sebelum giliran saya

🚦 Semaphore
Semaphore = variabel integer dengan dua operasi atomik:

wait(S) / P(S) — S--; kalau S < 0, proses tidur/block
signal(S) / V(S) — S++; kalau ada proses tidur, bangunkan satu

Jenis:
  • Binary Semaphore (0 atau 1) = mirip Mutex Lock
  • Counting Semaphore = bisa dipakai untuk sumber daya dengan banyak instance

📦 Bounded-Buffer Problem
Produsen mengisi buffer, Konsumen mengambil dari buffer. Buffer berukuran terbatas (N slot).

Semaphore yang dibutuhkan:
  • mutex = 1 — untuk mutual exclusion akses buffer
  • empty = N — slot kosong tersedia (produsen tunggu jika 0)
  • full = 0 — slot berisi (konsumen tunggu jika 0)
Buffer siap. Produsen mengisi, Konsumen mengambil.

07
Manajemen Memori
Address Binding, Paging, Segmentasi

Memori adalah "meja kerja" komputer. OS harus mengatur meja kerja ini agar semua program bisa bekerja tanpa saling mengganggu. 🗃️

📍 Address Binding
Pengikatan instruksi & data ke alamat memori fisik. Bisa terjadi di 3 waktu:
Compile Time
Alamat fisik ditentukan saat kompilasi. Jika alamat berubah, harus dikompilasi ulang.
Load Time
Alamat ditetapkan saat program dimuat ke memori. Bisa direlokasi tapi tidak saat berjalan.
Execution Time ⭐
Binding dilakukan saat runtime. Butuh MMU. Proses bisa dipindah-pindah di memori!
~ ~ ~
Logical Address (Virtual) = dihasilkan CPU. Yang program "lihat".
Physical Address = alamat sesungguhnya di hardware memori.
MMU (Memory Management Unit) = hardware yang menerjemahkan keduanya.

08
Memori Virtual
Demand Paging, Page Replacement, Thrashing

Virtual Memory = ilusi bahwa RAM kita sangat besar. Program merasa punya banyak memori, padahal OS cerdik memindahkan data antara RAM dan disk. 🎭

📥 Demand Paging
Page hanya dibawa ke memori fisik saat dibutuhkan (lazy swapping). Hemat memori!

Saat mengakses page yang tidak ada di memori → terjadi Page Fault:
1
Referensikan page → cek page table
2
Valid bit = 0 → Page Fault! Trap ke OS
3
OS cari page di backing store (disk)
4
Masukkan page ke frame kosong di RAM
5
Update page table (valid bit = 1)
6
Ulangi instruksi yang menyebabkan fault

🔁 Algoritma Penggantian Page
FIFO
Ganti page yang paling lama ada di memori. Sederhana. Bisa kena Anomali Belady (frame ↑, fault ↑).
Optimal ⭐
Ganti page yang tidak akan dipakai paling lama di masa depan. Terbaik tapi tidak bisa diimplementasi (butuh tahu masa depan).
LRU
Ganti page yang paling lama tidak dipakai. Approx Optimal. Bisa pakai stack atau counter untuk implementasi.
😵 Thrashing ⚠️ Penting!
Thrashing terjadi saat proses tidak punya cukup frame. OS terus swap halaman masuk-keluar. Utilisasi CPU jatuh drastis! Solusi: Working-Set Model — hanya jalankan proses yang "working set"-nya muat di RAM.
Thrashing = proses sibuk swap page keluar-masuk, tidak sempat kerja aslinya. CPU utilization anjlok!
Working-Set Model: pantau △ (delta) page terakhir yang digunakan. Working set = kumpulan page aktif dalam jendela waktu tersebut. Jika total WS > frame tersedia → suspend salah satu proses!
